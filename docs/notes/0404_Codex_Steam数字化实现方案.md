# Steam 数字化实现方案

这份方案的目标不是做一个“像桌游模拟器那样高度自由、组件到处可拖”的重型系统，而是做一个**轻量、规则优先、便于继续迭代**的单仓库小项目。因为这个仓库复现《Steam》的主要目的，是在实现和游玩中学习规则，所以程序设计应该优先服务三件事：第一，规则判断必须清楚且可验证；第二，界面要让玩家一眼看懂当前状态和可执行动作；第三，代码结构要足够薄，让后续继续补规则、补地图和补交互时不需要推倒重来。

这份文档现在以 `docs/references/rulebook_official.pdf` 作为主规范来源。`rulebook_scan.pdf`、`player_aid.pdf`、`quick_rules.doc` 和 `rule_summary.doc` 可以辅助理解，但当它们与官方规则书表达粒度不同，程序实现必须以官方规则书为准。

## 总体形式

我建议把这个项目做成一个**前后端一体的前端单页应用**，运行方式尽量简单：本地启动一个开发服务器，在浏览器里游玩和调试。技术选型上，最稳妥的轻量组合是 `Vite + TypeScript + React + SVG`。这里用 React 不是为了追求“大框架”，而是因为它可以很自然地把棋盘、侧栏、操作面板和弹层拆成状态驱动的组件，同时又不需要引入额外的状态管理库；状态保持在应用根部的 reducer 和少量 service 中即可。地图渲染优先用 SVG，而不是 canvas 或游戏引擎，因为 SVG 更适合六边形格、轨道高亮、所有权标记和可点击区域，也更容易在开发时直接调试 DOM。

如果后续你觉得 React 仍偏重，也可以退到 `TypeScript + Lit` 或纯 TypeScript 组件，但在当前阶段不建议一开始就为了“更轻”而放弃成熟的组件模型。对这个项目来说，真正会拖慢进度的不是框架体积，而是规则状态、交付判定和非法操作提示如果没有被组织好，后面每改一条规则都会牵一发而动全身。

## 目录组织

推荐把仓库保持成非常薄的一层文档、一层脚本、一层应用源码。核心是把“规则内核”和“界面壳”分开，让你以后既可以接 UI，也可以单独跑规则测试。

```text
steam-boardgame-replication/

│   ├── notes/
│   └── references/
├── scripts/
│   ├── git-hooks/
│   ├── export-map-data.ts
│   └── smoke-playthrough.ts
└── src/
    ├── app/
    │   ├── App.tsx
    │   ├── routes.ts
    │   └── boot.tsx
    ├── assets/
    │   └── maps/
    ├── core/
    │   ├── model/
    │   ├── rules/
    │   ├── engine/
    │   └── types/
    ├── data/
    │   ├── maps/
    │   ├── tiles/
    │   └── scenarios/
    ├── features/
    │   ├── setup/
    │   ├── actions/
    │   ├── build-track/
    │   ├── move-goods/
    │   ├── income/
    │   └── endgame/
    ├── ui/
    │   ├── board/
    │   ├── sidebar/
    │   ├── controls/
    │   ├── overlays/
    │   └── log/
    └── utils/
```

这里最重要的是 `src/core/`。`core/model/` 放最基础的数据结构，例如玩家、城市、城镇、hex、track tile、link、goods cube、action tile、turn state；`core/rules/` 放纯规则函数，例如“某块轨道能不能放在这个 hex”“这次交付是否合法”“本次建轨费用是多少”“终局如何计分”；`core/engine/` 放状态推进器，也就是把“执行一个动作”转换成“从旧状态得到新状态和日志”。只要这三层干净，`features/` 和 `ui/` 就能相对轻松地继续长大。

`src/features/` 不应该重复写规则，它只负责把一段用户意图组装成具体动作。例如 `build-track/` 负责“用户点了起点、点了一个 hex、选择了板块朝向”这件事如何形成一个 `BuildTrackAction`，然后把它交给引擎；`move-goods/` 负责“用户选中货物后，界面如何高亮允许目的地与路径候选”，真正的能否交付仍然由 `core/rules/` 判定。

`src/ui/` 只处理显示，不保存业务真相。比如棋盘上哪条线高亮、某个按钮是否灰掉、鼠标悬停提示是什么，这些都可以从当前游戏状态和一组候选动作推导出来，而不应该反过来让 UI 自己维护另一份“半真半假的状态”。

## 官方规则书带来的结构补充

官方规则书除了 Base Game 本体，还给了三个实现时非常重要、之前不能再省略的东西：`Tile Manifest`、`Cost Summary` 和完整的 Standard Game 差异。这意味着程序结构里至少要把“板块库存”“行动费用规则”“Base/Standard 规则切换”留出明确位置，而不是只靠文档记忆。

第一，`src/data/tiles/` 不应只存板块形状，还要存**官方板块配对与库存数量**。例如每种 tile face pair 有多少份、哪些是 town tile、哪些是 crossing tile，都应该数据化；这样建轨阶段才能正确判断“某类板块是否还有库存”，不会出现无限供应。

第二，`src/core/rules/` 里应该把费用规则做成显式表而不是散落在逻辑里。官方 `Cost Summary` 已经明确了基础版的建轨费用、`City Growth` 费用、`Locomotive` 费用和 `Urbanization` 费用，所以最合适的做法是实现一个 `ruleset` 配置对象，让 Base/Standard 两个模式共享相同引擎、只切换费用与阶段差异。

第三，如果你当前只打算先做 Base Game，也建议在 `src/core/types/` 里预留 `gameMode: "base" | "standard"`。因为官方规则书的 Standard Game 并不是扩展包式的“未来再说”，而是同一本规则书里的正式玩法；提前在类型层留接口，后面补 Standard 不需要拆核心状态。

## 规则内核的组织方式

这个项目最值得刻意设计的，是“动作驱动”的规则内核。不要让界面直接到处改状态，而是统一定义一套有限动作，例如 `SelectActionTile`、`PassOptionalAction`、`BuildTrackTile`、`PlaceNewCity`、`PlaceCityGrowth`、`UpgradeLocomotive`、`DeliverGoods`、`PassMove`、`ResolveIncome`、`AdvanceTurnOrder` 和 `StartNextTurn`。界面只发这些动作，所有状态变化都走同一个 `applyAction(state, action)` 入口。这样一来，规则测试、重放日志、撤销一步、回放整局和保存/读取局面都会变简单。

具体状态建议拆成四个块：`mapState` 记录地图上有哪些城市、城镇、地形、已放轨道、link 所有权和补货标记；`economyState` 记录玩家现金、收入、胜利点、机车等级与破产状态；`turnState` 记录当前阶段、当前轮次、行动顺序、行动牌选择结果和还剩几次运输动作；`supplyState` 记录货物袋、货物供应区、轨道板库存、新城市板库存和城市发展标记库存。拆开以后，每条规则都能落到比较稳定的对象上，不容易在实现时把“地图信息”和“回合临时状态”混在一起；其中 `supplyState.tilePool` 必须按官方 `Tile Manifest` 实现成有限库存，而不是无限数组。

针对基础版和标准版，建议把 `turnState` 再细分出一个 `phaseContext`。基础版里要支持“按需融资”和“初始顺位拍卖”，标准版里则要支持“每回合 Buy Capital”“每回合拍卖顺位”“机车维护费”“行动牌不再收费”。如果不把这些差异显式装进 phase context，后面很容易把 Base/Standard 逻辑硬搅在一起。

判定函数建议尽量做成纯函数，并返回详细结果，而不是只返回 `true` 或 `false`。例如建轨合法性函数不要只说“不能放”，而应返回类似“不能放，因为该 hex 是城市格”“不能放，因为轨道会穿过粗黑边”“不能放，因为你正在延伸他人的 link”。这个额外信息对数字化学习版尤其重要，因为它能直接转成界面上的提示文本，让玩家在非法操作时顺便学会规则。

此外，建轨规则至少要拆成四类纯函数：`canPlaceTrackTile`、`canRedirectIncompleteLink`、`canImproveExistingTrack` 和 `canClaimUnownedIncompleteLink`。官方规则书对这四件事给的是不同限制，尤其是“改向不算延伸”“升级必须保留旧线”“无主未完成连接只有特定玩家可接手”，如果合成一个大函数，后面很难查 bug。

## 界面设计

界面不应该追求桌游扫描件的完全复刻，而应该优先强化“状态可读性”。我建议做成三栏式布局。中间是主棋盘，占据绝大部分空间，用 SVG 画六边形地图、城市颜色、城镇、河流、轨道和玩家标记；左侧是全局状态栏，显示当前回合、阶段、行动顺序、货物供应区、行动牌可选项和公共库存；右侧是当前玩家面板，显示现金、收入、胜利点、机车等级、已选动作、当前可执行操作和操作日志。

棋盘交互应该尽量直接。鼠标悬停某个 hex 时，高亮地形、现有轨道和可放板块方向；点击货物时，自动高亮所有合法目的地和可能路径；点击一条现有未完成连接时，如果当前允许改向或延伸，就把可用选项高亮出来。由于游戏的学习目的很强，界面上最好一直存在一个“规则解释区”，专门显示系统为什么允许你做这步，或者为什么不允许你做这步。

视觉上建议明确区分三种信息层。第一层是固定地图信息，例如城市名称、颜色、地形、河流和粗黑边；第二层是可变网络信息，例如轨道板、link 所有权标记、新城市和城市发展标记；第三层是临时决策信息，例如高亮路径、候选落点、非法原因和本次交付的得分分配预览。只要这三层分得清，哪怕画面风格很朴素，玩家也会更容易理解局势。

## 最小可用版本

为了保持项目轻量，我建议把实现拆成三个很小的里程碑，而不是一开始就做联网、动画和多地图。

第一阶段只做“局面查看器 + 规则状态机”。它应该能载入一张地图 JSON，展示城市、城镇、地形和轨道库存，并支持手动推进回合、选行动牌、建轨和结算费用。这一阶段即使还不能完整游玩，也已经能验证建轨规则、费用规则和回合流程是否正确。

第二阶段补“货物运输与计分”。这时要加入交付路径判定、机车等级限制、自己用路不少于任一单个对手的限制、交付后收入/胜利点二选一，以及终局计分。做到这里，游戏就已经可以作为规则学习工具使用。

第三阶段再补“体验层”。例如操作日志、一步撤销、保存/读档、教学提示、开局随机补货、示例剧本和更顺手的高亮。这些都很有价值，但应该建立在规则内核已经稳定的前提上。

## 数据格式

地图和板块数据建议完全数据化，不要硬编码在组件里。地图 JSON 至少需要描述 hex 坐标、地形类型、是否有河流、哪些边不可通行、城市/城镇信息、城市颜色和初始货物数量；轨道板数据需要描述板块类型、出口边集合、是否为城镇轨道、是否为复杂交叉，以及基础费用。只要数据格式先统一，以后增加别的地图、扩展板块或不同玩家人数的设置时，基本不需要改核心引擎。

这里还应补一层 `ruleset` 数据。它至少要包含：`turnCountByPlayerCount`、`baseActionCosts`、`standardAuctionPaymentRule`、`locomotiveMaintenanceRule`、`canRaiseMoneyDuringTurn`、`cityGrowthHasPassOption`、`urbanizationHasPassOption` 等。这样你在实现基础版时就不会把“规则文本中的常量”硬写死在 reducer 里。

地图数据也不该只停留在“视觉坐标”。为了支持交付和 link 完成判定，建议同时维护一个**六边格邻接层**和一个**铁路图层**：前者记录 hex 与六方向边界属性，后者记录当前哪些出口已经被轨道连通、这些连通归谁所有、是否构成完整 link。只有这两层都在，程序才能既正确判断“轨道能不能放”，又正确判断“货物能不能走”。

货物颜色、行动牌类型、阶段枚举和玩家颜色也都建议做成有限枚举并集中管理，而不是散落字符串。这样既方便 TypeScript 收窄类型，也能避免以后界面层和规则层出现“拼写接近但不相等”的小错误。

## scripts 的职责

`scripts/` 保持非常克制，只放少量辅助脚本，不要把业务代码塞进去。`export-map-data.ts` 用于把手工整理的地图资料转成程序消费的 JSON；`smoke-playthrough.ts` 用于跑一段固定动作序列，检查核心规则在 CLI 下能否从开局顺利推进到若干轮之后。这样做的好处是，你既有浏览器里的可视化调试，也有一个很轻量的命令行“冒烟验证”入口。

基于官方规则书，我建议再加一个 `scripts/check-tile-pool.ts`，专门校验 `src/data/tiles/` 里的板块数量是否和官方 `Tile Manifest` 一致。这个脚本很小，但很值，因为一旦 tile pool 配错，游戏后期的建轨合法性就会全部偏掉。

如果后续需要测试，不建议一开始就搞很重的测试体系。一个现实的轻量方案是先给 `core/rules/` 和 `core/engine/` 写少量 Vitest 单测，覆盖最容易错的地方，例如建轨非法性、未完成连接所有权、城市化补货、货物交付路径限制和终局计分；同时用 `smoke-playthrough.ts` 保留一两条完整回合剧本，确保大流程没有被改坏。

优先单测里还应该额外加入三组：初始顺位拍卖与找零、有限 tile pool 耗尽后的非法建轨、升级既有轨道时“必须保留旧线”的约束。这三组都是官方规则书有明文、但程序很容易写偏的地方。

## 规则书留白与程序判定方案

+ `City Growth` 或 `Urbanization` 需要从 `Goods Supply` 取一整组 cubes，但官方规则书没有写当所有 supply 组都已经被拿空时，这两个动作还能否选择。我的建议是：动作仍可选择且仍可执行，但放出的城市不获得额外 cubes，同时在日志中明确标记为“supply depleted”。
+ 官方规则书没有明确写“当所有印刷城市都已有 `City Growth marker` 时，`City Growth` 是否仍可被选择”。我的建议是：允许选择，并保留原规则里的 `pass option`；若执行时不存在合法目标城市，则该动作只能 `pass`，不能强行落到非法城市。
+ 破产后留下的“无主已完成 link”在交付时不为任何人得分，但规则书没有明确它在“你使用自己 link 的数量必须不少于任何单个其他玩家”这一限制里该怎么算。我的建议是：把无主 link 视为 `owner = null` 的中立 link，允许经过，但它不计入任何玩家的配额比较，也不产生分数。
+ 官方规则书没有明确写无主已完成 link 之后能否被别人继续“improve existing track”。我的建议是：允许在不破坏既有轨道走向的前提下进行升级，因为 `improve existing track` 的限制核心在于保留旧线，而不是旧线是否仍有玩家所有权。
+ 规则书没有明说当 `Urbanization` 让一个新城市瞬间补全多个玩家的未完成连接时，若其中有人本回合稍后还会行动，这些 link 的完成时点与得分资格是否立即生效。我的建议是：新城市一落下就立刻更新 link 完成状态，从那一刻起这些 link 已可被后续交付使用，因为官方规则已明确新城市可能完成其他玩家的 link。
+ 规则书没有给出“教学型数字界面”所需的唯一选路规则；当同一玩家能用不同合法路径把同一个 cube 送到同一个终点时，桌面上通常由玩家自己声明路径。我的建议是：界面显式要求玩家点选完整路径，并在提交前预览每位玩家将获得的 track points，而不是试图偷偷自动选最优路。

## 实现顺序建议

最先落地的文件不应该是复杂 UI，而应该是 `types`、地图 JSON 和 `applyAction` 所需的最小状态结构。接着实现“选行动牌”和“建轨”的状态推进，因为这两块最能暴露地图表示是否合理；再实现运输和计分，因为它们对 link 数据结构要求最高；最后才是更顺手的高亮、日志和视觉修饰。

如果你只想先快速做出一个能玩的版本，那么第一版甚至可以不做拖拽，只做点击式交互：点击 hex 选目标，右侧面板选择轨道板和朝向，系统即时提示是否合法。这样开发量比自由拖拽小很多，但对学习规则已经足够有效。

## 结论

这个项目最合适的方向，不是“桌游模拟器”式的大而全，而是“规则引擎 + 教学型可视化界面”的小而稳。只要 `src/core/` 里的状态和动作设计清楚，`src/ui/` 保持对规则反馈友好，再配上两三个很轻的 `scripts/` 辅助脚本，你就能在不引入重型技术栈的前提下，把《Steam》做成一个真正能边玩边学规则的数字化版本。
