# Steam 第二部分实现方案及日志

这部分要解决的问题，已经不再是“先让系统动起来”，而是要把第一部分的演示壳推进成一个**可以由真人实际操作、并能明显帮助理解规则的基础版可玩原型**。第一部分已经证明当前仓库的状态树、阶段机、草稿层、最弱 Bot 和最小测试闭环都能成立；第二部分的任务，是把这些底层能力真正接到用户操作和规则反馈上，让程序从“自动演示状态变化”升级为“用户可以自己做决定，系统能即时告诉他能做什么、为什么、结果会怎样”。

因此，第二部分不追求一次写完《Steam》基础版的全部边角，也不急着做标准版，更不应先做漂亮但空心的地图交互。它的核心交付应当是：**一名真人玩家在本地浏览器里控制自己的阵营，其余座位由 Bot 托管，完整走完基础版的大部分主流程，并在每个关键决策点看到合法选项、非法原因和结果预览。**

## 第二部分的交付目标

第二部分结束时，仓库应当具备以下能力。第一，玩家可以在界面里显式选择行动牌，而不是只能靠自动推进。第二，玩家可以在建轨阶段点击地图上的合法位置，选择轨道板和朝向，并在提交前看到成本、融资结果和草稿中的线路变化。第三，玩家可以在运输阶段看到系统搜索出来的合法候选方案，并在候选方案之间选择，而不是被动接受随机 Bot 的决定。第四，公共信息区要开始像真正的桌游桌面那样工作，至少清楚展示行动牌占用情况、Goods Supply、轨道板库存摘要、回合/顺位和当前玩家状态。第五，测试必须扩展到“用户驱动的主流程”，保证第二部分新增的交互不是只在页面上能点，而是真正接在规则引擎上。

这意味着第二部分完成后，浏览器里的主要操作应当已经从“自动执行一步 / 自动执行十步”切换成“玩家主动选择，Bot 只负责其他玩家或补足自动测试”。自动推进按钮可以保留，但它只能作为调试工具，不能再是主交互。

## 为什么第二部分从这里开始

第一部分已经把底层地基立住了。当前仓库已经有 `createGame.ts`、`phaseMachine.ts`、`draftSession.ts`、`applyAction.ts`、`goodsDelivery.ts`、`trackPlacement.ts`、`RandomBot.ts` 和一套基础测试，所以继续追加“更多自动化逻辑”并不能真正提升可用性。当前最明显的瓶颈其实有三类：一是动作模型还偏粗，很多阶段只有“这个阶段可以做什么”的抽象选项，没有“当前局面下具体能点哪些对象”的细粒度数据；二是 UI 仍然是演示页，地图只是展示层，几乎没有把用户输入接回引擎；三是规则解释虽然有提示文本，但还没有形成“当前点击对象 -> 校验 -> 预览 -> 执行结果”的完整反馈链。

第二部分的重点正好就是补这三条链路。只要这一步补上，后面无论是标准版、更多地图、存档、教程模式还是更强 AI，都会站在一套真实可玩的基础版界面上继续长；如果不先把这一层接起来，后续扩展仍然会停留在“代码里有规则，但玩家在界面上用不到”的状态。

## 第二部分的完成标准

第二部分完成时，应以“真人 1 人 + Bot 若干”的常见使用方式来衡量，而不是以纯命令行或纯 Bot 自跑来衡量。你应当能够启动本地页面，选择自己的玩家颜色，用鼠标和面板操作完成以下流程：选择行动牌；在建轨阶段选择轨道板并点击地图落位；在货运阶段查看候选运货方案并执行其一，或者选择升级机车 / 跳过；观察收入、顺位和下一回合的推进；直到进入后续回合并保持状态、日志和公共信息一致。如果没有达到这个标准，第二部分就还没有真正完成。

## 当前状态与第二部分的真实缺口

现在的仓库已经适合接第二部分，但也暴露出很明确的缺口。`src/state/actionTypes.ts` 中的动作仍然偏向引擎内部，缺少“开始选择轨道板”“更新当前高亮 hex”“选择运输候选方案”“确认当前预览动作”这类更贴近界面的动作。`src/engine/selectors.ts` 目前能给出阶段级 `ActionOption`，但无法告诉 UI “当前有哪些可选行动牌”“当前哪些 hex 合法”“某个 hex 上允许哪些 tile 和 rotation”“当前有哪些可选货物”“当前有哪些 ranked candidates”。`src/ui/MapBoard.tsx` 仍是简化 SVG，可视化的是每个 hex 的货物数和轨道数，并不支持点击落子；`src/ui/PhasePanel.tsx` 仍然主要服务于自动推进；`src/engine/applyAction.ts` 在运货结算上仍是第一部分口径，更多是为了闭环，而不是为了完整表达基础版的执行细节。

因此，第二部分不是“继续在现有页面上加几个按钮”就够了。它需要在引擎层、查询层和 UI 层同时扩展一轮，而且这轮扩展必须围绕“一个真人玩家如何完成实际决策”来设计。

## 第二部分的核心原则

第二部分的设计应当坚持四个原则。第一，**动作必须可视化且可回退**。玩家在建轨和运货阶段看到的不是一团不可解释的状态，而是一组当前可以做、正在预览、提交后会发生什么的对象。第二，**合法性搜索必须先于交互**。地图点击不能靠“点了再试试”，而要由程序先告诉前端哪里可点、点后还能选什么。第三，**主界面必须开始区分公共区、个人决策区和地图区**，否则规则学习会一直混在一块。第四，**测试要优先锁住用户主路径**，不能只保留规则单测而缺少“界面动作 -> 引擎动作 -> 状态变化”的集成验证。

## 第二部分的任务拆分

第二部分建议拆成五个相互衔接的任务块。这个拆法的目的不是把工作切碎，而是把当前最阻塞“真人可操作”的环节排出顺序，使每一块都能产出可验证结果。

### 任务块一：把动作模型升级成面向交互的动作与查询

这一块只改引擎接口，不急着碰地图点击。当前 `GameAction` 需要从“最少能跑通的动作联合”升级成“适合 UI 驱动的显式动作集合”。推荐新增两类内容：一类是**面向状态修改的正式动作**，例如 `select-human-player`、`preview-track-placement`、`confirm-track-placement`、`select-delivery-candidate`、`confirm-delivery`、`cancel-current-preview`；另一类不一定要进入 `GameAction`，但必须在 `selectors.ts` 中以只读查询形式提供，例如“当前可选行动牌列表”“当前可点 hex 列表”“给定 hex 的合法 tile/rotation 列表”“当前货物源列表”“给定源货物的 ranked delivery candidates”。

这里的关键不是把所有 UI 临时状态都塞进 `gameState`，而是要区分三层：正式局面仍然放在 `gameState`；阶段级可提交内容仍然放在 `draftState`；纯展示用的选择态，例如“当前 UI 正在看哪一个 hex、哪一个 tile、哪一个运输候选”，应当放在 React 组件状态或单独的轻量 UI store 中。这样规则引擎依然保持干净，UI 也不会因为每次 hover 或切换面板而污染历史状态。

建议在这一块直接补两个新查询模块：`src/engine/legalMoves.ts` 负责聚合当前局面的可执行具体对象，`src/engine/previews.ts` 负责把“如果执行这个动作，会发生什么”整理成 UI 可渲染的摘要。第二部分结束时，UI 不应再自己拼规则逻辑；它应该只消费这些明确的查询结果。

### 任务块二：把建轨阶段做成真正可交互的草稿工作台

建轨阶段是第二部分里最重要的一块，因为它同时覆盖地图点击、有限板块库存、融资、所有权重建和草稿机制。这里的目标不是一次做出完美拖拽，而是做出一条可靠的工作流：玩家先在侧栏选择待铺设的轨道板类型，再在地图上点选合法 hex，系统根据该 hex 和当前轨网给出合法朝向；玩家选中具体方案后，在右侧看到成本、是否会触发融资、是否会新开 link、是否会影响锚点，然后点击确认，把这一步写入当前阶段草稿。若玩家不满意，可以继续追加、重置本阶段或最终提交。

要做到这一点，`trackPlacement.ts` 当前返回的“能不能放”和“费用是多少”还不够，需要再补一层更适合界面消费的结构化校验结果，例如：合法出口集合、阻塞原因分组、是否要求城镇轨道、是否属于升级现有轨道、是否会连接到哪些城市。与之对应，地图层也不能只渲染单个 polygon；它需要知道当前哪些 hex 是可交互的，当前选中 hex 的可用 rotation 是哪些，以及草稿中的新轨道与正式轨道如何区分颜色或描边。

这一块完成后，当前的“自动执行一步”应该仍能工作，但建轨阶段必须已经可以由真人手动完成，不再依赖 Bot 才能往下走。

### 任务块三：把货运阶段做成“源货物 -> 候选方案 -> 结果预览”的流程

第一部分已经有 `goodsDelivery.ts` 和 `routeRanking.ts`，但它们主要是为测试和最小闭环服务。第二部分要把这部分真正翻译成用户可以理解和操作的流程。推荐的交互顺序是：玩家先在地图或侧栏里选择一个可运输的货物源；程序立刻调用 `routeSearch.ts` 搜索合法路径，再由 `routeRanking.ts` 整理出少量候选方案；界面用单独的候选面板列出每个方案的目的地、长度、借道情况、各玩家得分分配、机车需求和规则备注；玩家点选其中一个方案后，地图高亮路径，右侧预览区显示执行后的分数变化和货物消耗；玩家确认后，再由引擎真正执行 `deliver-goods`。

这一块的关键不是“自动找最优路线”，而是“系统算出合法候选，玩家理解后自己选”。如果当前局面下没有任何合法运输方案，界面要明确显示“无合法运输方案”的原因，并把升级机车和跳过作为当前阶段的主要替代动作。这样运输阶段才会像教学工具，而不是黑盒自动运算器。

同时，这一块也是第二部分里最适合把规则解释真正做起来的地方。每个候选方案都应附带一段短解释，例如“经过自己 2 段、对手 A 1 段、对手 B 0 段，因此满足借道比例限制”，或者“在首个蓝色城市必须停止，因此不可继续延长到更远城市”。这些解释应该来自规则函数输出，而不是 UI 自己事后拼接。

### 任务块四：把公共区和玩家区补成真正可读的桌面信息面板

当真人开始操作后，当前右侧那种“阶段 + 玩家 + 提示 + 日志”的最简仪表盘就不够了，因为它缺少桌游里最重要的公共信息。第二部分要把公共区补出至少四个独立面板：行动牌面板、Goods Supply 面板、轨道库存摘要面板、玩家状态面板。行动牌面板需要显示已选 / 未选、当前回合谁先选、哪些牌已被占用；Goods Supply 面板需要显示六组 supply cubes 和货物袋剩余概况；轨道库存摘要面板至少要按 tile 类别显示剩余量，而不是只在后台扣减；玩家状态面板除了现金、收入、VP、机车外，还应开始显示本回合已选行动牌和是否已升级机车。

这一块的价值，在于把“程序内部已经知道的信息”真正摆到桌面上，让玩家不用翻日志猜当前局面。实现上可以新增 `ActionTilePanel.tsx`、`GoodsSupplyPanel.tsx`、`TilePoolPanel.tsx` 和更完整的 `TurnOrderPanel.tsx`；也可以先合并其中两块，但不要继续把所有信息塞进 `PhasePanel.tsx` 一张卡里。

### 任务块五：把测试从规则单测扩到用户主路径集成测试

第二部分引入交互后，原有单测仍然重要，但已经不够。除了保留 `finance`、`tilePool`、`trackPlacement`、`tokenAnchors`、`goodsDelivery`、`routeRanking`、`scoring` 和 `bot` 这些基础测试，还应补一批专门验证“用户主路径”的集成测试。推荐新增四组：`humanActionSelection.test.ts`，验证真人玩家可以在选牌阶段选择合法行动牌且 UI 查询结果正确同步；`buildDraftFlow.test.ts`，验证建轨预览、确认、重置、提交这条链路不会污染 committed state；`deliveryWorkflow.test.ts`，验证源货物选择、候选方案排序、预览和确认执行的一致性；`uiShell.test.tsx`，验证关键面板在典型局面下能正确展示状态和禁用非法按钮。

这里不要求第二部分就写大量浏览器端端到端测试，但至少要把“状态选择器 + 交互查询 + 关键组件渲染”这层锁住。否则第二部分一旦开始铺 UI，回归风险会迅速变大。

## 第二部分的实现顺序

第二部分的实现顺序应当严格服务于“尽早让真人可操作”，而不是平均推进所有子系统。推荐按下面六步做。

### 1. 先扩查询层，再动 UI

先把 `selectors.ts` 拆出一部分给 `legalMoves.ts` 和 `previews.ts`，让 UI 在不理解规则细节的前提下也能拿到结构化数据。比如 `getSelectableActionTiles`、`getBuildableHexes`、`getTrackPlacementOptionsForHex`、`getMovableGoodsSources`、`getRankedDeliveryCandidatesForSource`、`getTrackPlacementPreview` 和 `getDeliveryPreview`。这一层完成后，再开始做新面板，否则 UI 很容易重新把规则逻辑写一遍。

### 2. 把选牌阶段改成真人可操作

当前最容易接的是真人选牌。先新增 `ActionTilePanel.tsx`，把选牌阶段从“自动推进”改为“真人可选、Bot 自动补齐其他玩家”。这一步完成后，页面就会第一次从纯演示壳变成“真人能实际影响局面”的界面。

### 3. 再把建轨工作台做出来

选牌一旦能手动完成，接下来应立即做建轨工作台，因为它是第二部分里工作量最大、价值也最大的功能。这里要同时完成 Tile 选择、地图 hex 选择、rotation 选择、预览、确认、重置和提交。只有把建轨工作台跑通，草稿层的价值才真正落到玩家视角。

### 4. 接着补货运候选面板和预览

建轨完成后，再把货运阶段改成“选择源货物 -> 查看候选方案 -> 预览 -> 执行”。这一块建议优先完成单货物源的完整流程，再补地图高亮与更漂亮的候选卡片。

### 5. 之后再补公共区面板与更完整的桌面信息

这一步不是最早做，但必须在第二部分末尾补上。原因很简单：没有公共区信息，玩家虽然能点，但仍然很难理解局面和规则代价。第二部分至少要让玩家能直接看到行动牌占用、轨道库存、货物供应和回合顺序。

### 6. 最后做一轮集成测试和界面可读性整理

当主要功能接上后，再统一补集成测试，并做一轮界面收口。这里要检查的不只是测试是否通过，还包括：典型 16:10 窗口里是否能在一屏内看到地图、当前行动区和最近日志；非法按钮是否被禁用；日志、提示和候选解释是否用了足够简短直接的中文。

## 第二部分的状态设计补充

第二部分里最容易出问题的点，是把所有“正在选什么”的临时信息都塞进引擎状态。这里推荐明确区分三类状态。第一类是**规则真相**，例如轨道、货物、现金、收入、顺位、当前阶段，这些必须进入 `gameState` 或 `draftState`。第二类是**阶段内待确认的规则操作**，例如建轨草稿里已追加但尚未提交的轨道、当前阶段已消耗的建轨额度，这些继续放在 `draftState`。第三类是**纯界面选择态**，例如当前高亮哪个 hex、当前选择哪个 tile、当前正在查看哪个运输候选方案，这些不应进入规则引擎，而应留在组件状态中。

如果把第三类状态也塞进引擎，后面日志、重放、测试和 Bot 都会被 UI 噪声污染；如果把第二类状态也丢给组件，又会破坏草稿机制。第二部分应当开始把这个边界收得很明确。

## 第二部分的规则实现重点

第二部分虽然还不要求补齐全部边角规则，但有几条必须从“第一部分的最小闭环口径”升级到“接近真实基础版口径”。第一，运货执行不能只更新当前玩家自己的分数；它需要按照候选方案中各所有者的收益分配去更新分数或至少保留完整收益记录。第二，机车升级、跳过货运和两轮货运之间的关系必须更准确地反映当前玩家在该轮真正可选的行为。第三，建轨合法性检查要开始把更细的非法原因暴露出来，例如“城镇 hex 只能使用城镇板”“当前 tile 库存为 0”“会切断既有走向”“不能跨越粗黑边”。第四，顺位与行动牌占用状态必须从后台逻辑真正反映到 UI，而不只是日志里提一句。

第二部分结束时，规则层仍然可能有留白，例如官方规则书未明确指定的极端边界局面；但对于基础版正常对局中最常见的主路径，程序不应再依赖推测性默认行为。

## 第二部分的测试标准

第二部分的测试标准应当比第一部分更接近真实使用。最低要求是：

+ 一个真人玩家能在浏览器里手动完成至少两个完整回合的主要决策，另外两名 Bot 不会卡死。
+ 建轨阶段的预览、确认、重置、提交四条链路都能在测试中验证 committed / draft 两层状态是否正确分离。
+ 运输阶段至少能覆盖“有合法方案”“无合法方案”“升级机车替代运货”“跳过本轮”四种常见情况。
+ UI 主要面板在 16:10 左右窗口中不依赖频繁滚动，关键信息始终可见。
+ `npm test`、`npm run build` 和 `npm run smoke` 必须继续通过。

## 第二部分结束后的仓库状态

如果第二部分按上面的目标完成，仓库应当从“第一部分的规则演示壳”进入“基础版可玩原型”阶段。到那时，页面的主按钮应当以真人操作为主，自动推进退到调试位置；地图应当开始承担真实交互；候选方案与预览应当成为学习规则的主要抓手；文档里写的“教学型界面”才算真正落地。后续第三部分才适合继续处理完整地图细节、更多规则边角、标准版差异、存档回放和更系统化的教学模式。

## 结论

第二部分的正确目标，不是继续堆更多底层模块，也不是先做漂亮棋盘，而是把第一部分已经存在的引擎能力真正翻译成**真人可操作、规则可解释、状态可预览**的基础版原型。最重要的工作顺序是：先扩查询层，再做选牌，再做建轨工作台，再做货运候选与预览，最后补公共区和集成测试。只要这条线走通，仓库就会从“能动”升级到“能用”，也才真正接近这个项目最初的目标。
