# Steam 当前实现系统说明

这份文档描述的是仓库**当前真实已经实现**的系统，而不是最初方案里希望最终达到的完整形态。先给出结论：**现在的项目已经是一个可运行、可测试、可教学演示的《Steam》数字化原型，但还不能算“已经完全按照规则书写出了完整电子版 Steam”**。它已经实现了一套可以启动、本地运行、推进主要回合流程、支持 1 真人 + 若干 Bot、支持建轨草稿、货运候选、规则解释、存档与回放的学习型系统；但它仍然缺少若干规则书中的主行动和完整收口项，因此不能把“主体已成形”写成“已经完整复现”。

造成这个判断的关键原因很直接：当前动作系统里已经实现的是“选行动牌、建轨、运货、升级机车、收入结算、回合推进”这一条主路径，但还没有把 `City Growth`、`Urbanization` 以及它们牵出的新城市放置、城市发展补货、相关边界判定完整接进引擎动作层和页面交互层。标准版目前也主要停留在规则集和建局配置层面，还没有达到“基础版、标准版都能完整打一局”的程度。因此，如果按 [0404_Codex_Steam数字化实现方案.md](0404_Codex_Steam数字化实现方案.md) 的最强表述来核对，答案是：**还没有全部完成；但已经完成了其中最核心、最可运行、最有教学价值的一大部分。**

下面这份说明文档的目标，不是给出愿景，而是让读者透彻理解：现在的仓库里已经有哪些模块，它们如何分工，它们之间如何串联，以及当前版本到底能做什么、不能做什么。

## 一、当前实现的整体结构

从运行形态上看，这个项目是一个**本地浏览器单页应用 + 纯 TypeScript 规则内核 + 静态规则内容包 + 工具脚本层**的工作区。用户平时真正接触的是 `apps/web` 提供的本地页面；页面本身不直接写规则，而是通过 `packages/game-core` 暴露的引擎 API 读写状态；地图、行动牌、tile manifest、规则集等静态内容则由 `packages/game-content` 注入；测试、smoke、边界检查、数据核对脚本和 git hook 则集中放在 `tools`。

这套结构的核心思想是把“规则”和“界面”拆开。也就是说，浏览器负责展示棋盘、面板、按钮和提示，但它本身不应该决定一条路能不能走、一块轨道能不能放、一次运货为什么合法；这些判断都应该先在 `game-core` 中得出，再由 `web` 负责把结果翻译成当前页面需要的显示形式。当前仓库已经基本收到了这个边界上。

可以把当前系统理解成四层：

+ `packages/game-content` 负责回答“这局游戏使用什么内容”，也就是地图、规则集、行动牌、货物袋和轨道库存长什么样。
+ `packages/game-core` 负责回答“在这些规则和内容下，当前局面是什么、某个动作是否合法、执行后会变成什么样”。
+ `apps/web` 负责回答“把这些状态和规则结果如何呈现在浏览器里，以及让真人玩家如何发出动作”。
+ `tools` 负责回答“如何验证、构建、约束和辅助维护这个系统”。

## 二、当前版本到底实现到了什么程度

如果从“用户能在页面里直接做什么”来描述，当前版本已经支持下面这条真实可走通的主路径：

+ 在本地浏览器里创建一局默认的基础版东北美洲学习局。
+ 由第一个玩家作为真人，其余玩家由 `RandomBot` 托管。
+ 在行动牌阶段为真人选择行动牌。
+ 在建轨阶段选择轨道板、选择地图 hex、选择朝向，并通过草稿机制反复试走、提交或重置本阶段。
+ 在货运阶段选择货物源，并在系统给出的候选运输方案中选择一种合法方案执行。
+ 在合适阶段升级机车、跳过货运、推进收入结算和回合推进。
+ 在页面上实时查看规则解释、非法原因、公共信息、玩家状态、动作历史和日志。
+ 把当前局面保存到浏览器本地存储，重新载入，并通过回放帧恢复到较早时刻。

如果从“规则覆盖面”来描述，当前版本已经有这些比较扎实的基础：

+ 有明确的回合阶段机。
+ 有建轨成本、融资补钱、tile pool 扣减、轨道所有权重建、货运合法性搜索、候选运输排序、收入结算和终局阶段框架。
+ 有 `committed + draft` 双层状态，用来支持建轨阶段的整体回滚。
+ 有 `segment + anchor + link` 这一套比简单“线路数组”更稳的所有权模型。
+ 有 Web 层的规则解释 presenter，而不是把展示文本全部糊在 core 里。
+ 有带 schema version 和 migration 的序列化层。
+ 有规则测试、工作流测试、不变量测试、存档测试、页面测试，以及 `typecheck / lint / deps / boundaries / build / smoke` 这整条验证链。

但如果从“是否已完整实现规则书中的 Steam 基础版与标准版”来问，当前答案仍然是否定的。当前至少还有这些明显缺口：

+ `GameAction` 里还没有 `City Growth`、`Urbanization` 等动作，也就意味着这两类主行动还没有接成用户可走的正式流程。
+ 当前新对局面板明确写的是“重开基础版东北美洲对局”，也就是交互入口仍然固定在基础版东北美洲，而不是已经开放基础版/标准版、东北美洲/Ruhr 的完整切换。
+ 标准版目前已经有 `ruleset` 和建局配置测试，但还没有形成“与基础版并列、能完整打一局”的页面工作流。
+ 当前系统更像“学习壳 + 规则工作台”，还不是一套把整本规则书都封闭起来的最终电子桌游。

因此，更准确的说法是：**它已经完成了总设计里最重要的内核、工作流和教学架构，但还没有完成“规则书完整闭合”的最终收口。**

## 三、仓库主干与包级职责

当前仓库的核心主干如下：

```text
steam-boardgame-replication/
├── apps/
│   └── web/
├── packages/
│   ├── game-core/
│   └── game-content/
├── tools/
├── docs/
└── package.json
```

这里最重要的不是树本身，而是依赖方向：

+ `apps/web` 可以依赖 `@steam/game-core` 和 `@steam/game-content`。
+ `packages/game-content` 可以依赖 `@steam/game-core` 的类型契约，但不应反向依赖 Web。
+ `packages/game-core` 不依赖浏览器或 React，只负责纯规则和状态。
+ `tools` 只做辅助，不应侵入运行时规则逻辑。

这个依赖方向不是只靠约定维持的。当前仓库已经有 `eslint` 的 import 限制、`dependency-cruiser` 的依赖方向检查、自定义 `check-import-boundaries.ts`，以及 CI 中独立的 `typecheck / lint / check:deps / check:boundaries / test / build` 任务来把边界锁住。

## 四、`packages/game-content`：静态规则内容层

`packages/game-content` 的职责很单纯：它不推进游戏状态，也不判断动作是否合法，它只提供“这局使用的内容”。

这一层当前主要包含四类数据：

+ 地图数据：例如 `src/maps/ne_usa_se_canada.ts` 和 `src/maps/ruhr.ts`。
+ 规则集数据：例如 `src/rulesets/base.ts` 和 `src/rulesets/standard.ts`。
+ 开局与公共区静态内容：例如行动牌、货物袋组成、Goods Supply 参数、新城市颜色池。
+ 轨道板库存与形状数据：例如 `src/tiles/manifest.ts`。

这一层的组织逻辑，是把“可变的运行时状态”和“静态的规则内容”分开。地图 hex、行动牌定义、tile 数量这些内容是注入给引擎的，不是引擎自己硬编码生成的。这样做有两个直接好处：第一，基础版和标准版可以共享同一套运行时结构，只替换规则集；第二，未来增加新地图或修正某张地图的数据时，不需要改引擎代码本身。

当前 `src/index.ts` 提供了两个关键入口：一个是 `steamContentCatalogs`，把行动牌、tile manifest、货物袋组成、Goods Supply 参数和新城市板颜色组装成一份内容目录；另一个是 `getMapDefinition()` 与 `getRuleSet()`，让上层通过 `mapId` 和 `mode` 取到对应地图和规则集。这意味着内容包已经从“若干零散常量”升级成了“可以被建局流程正式消费”的包级 API。

但要注意，这一层虽然已经放入了 `ruhr` 和 `standard` 的入口，不代表它们在当前产品形态中都已经完全开放给用户使用。它们在“静态内容存在”这个层面是有的，在“完整可玩工作流已经接通”这个层面还没有完全到位。

## 五、`packages/game-core`：规则引擎与状态核心

`game-core` 是当前仓库最重要的部分。它承担的不是“帮页面算几个数字”，而是整个 Steam 运行逻辑的核心真相。

### 5.1 状态总模型

当前状态模型的中心类型在 `src/state/gameState.ts` 与 `src/contracts/engine.ts`。

`GameState` 描述的是一局游戏在某个时刻的正式局面，它被拆成几块：

+ `players`：每个玩家的现金、收入、胜利点、机车等级、是否破产、是否 Bot。
+ `map`：地图定义、已铺轨道、segment、anchors、links、新城市、城市发展标记和城市货物。
+ `supply`：货物袋、Goods Supply、轨道库存、新城市板池。
+ `turn`：当前回合、最终回合、阶段、顺位、行动牌选择、建轨剩余额度、货运次数、机车升级记录。
+ `logs`：动作和阶段日志。

`EngineSession` 在 `GameState` 之上再包了一层，它额外持有：

+ `committed`：已经正式写入历史的状态。
+ `draft`：当前阶段的草稿状态。
+ `config`：建局时的 `mode / mapId / playerCount / humanPlayerIndex`。
+ `actionHistory`：结构化动作历史。

这个设计是当前仓库最值得保留的一个决定，因为它正好对应《Steam》这种桌游的真实决策结构。特别是在建轨阶段，玩家往往不是单步撤销，而是想“回到本阶段开始时重新铺一遍”。`committed + draft` 就是为这个场景服务的。

### 5.2 建局与初始状态

建局入口是 `src/actions/createGame.ts`。它接收玩家名、Bot 玩家 ID、地图、规则集和内容目录，然后调用 `createInitialState()` 创建正式状态，并把这些内容收进 `EngineSession`。

这里的关键点不在“把对象拼起来”，而在于建局时就完成了三件重要绑定：

+ 这局使用哪张地图。
+ 这局使用哪套规则集。
+ 这局使用哪份共享内容目录。

也就是说，引擎不会自己去“猜”基础版还是标准版，也不会默认只有一张地图，而是由上层显式把内容注入进去。这个结构对后续扩展非常重要。

### 5.3 动作分派与 handler 架构

当前动作总入口是 `src/actions/applyAction.ts`。它的职责很克制：先读取当前工作态，校验是不是当前行动玩家，然后把不同动作分派给各自的 handler，最后把结果写回 `EngineSession` 并追加动作历史。

目前已经接入的显式动作包括：

+ `select-action-tile`
+ `place-track`
+ `finish-build`
+ `deliver-goods`
+ `upgrade-locomotive`
+ `pass-move`
+ `resolve-income`
+ `advance-turn-order`
+ `set-up-next-turn`

这份动作列表本身就说明了当前产品形态：行动牌选择、建轨、运货、收入和回合推进已经接上，`City Growth` 和 `Urbanization` 还没有正式进入动作层。

当前 handler 已经不是一个巨大总文件，而是按职责拆到了 `src/actions/handlers/` 下。比如：

+ `selectActionTile.ts` 负责行动牌选择和从选牌阶段进入建轨阶段。
+ `placeTrack.ts` 负责建轨草稿中的单步落子。
+ `moveGoods.ts` 负责运货、升级机车和跳过货运。
+ `phaseFlow.ts` 与 `phaseProgress.ts` 负责阶段推进和收入结算。
+ `shared.ts` 负责日志、历史、越权动作警告等共用逻辑。

这套拆法的价值，在于把“规则执行”和“会话写回”分开了。`applyAction` 不再自己知道建轨成本怎么算、运货是否合法，而只是承担调度器角色。

### 5.4 阶段草稿机制

阶段草稿由 `src/actions/draftSession.ts` 管理。它提供 `beginDraft()`、`ensureDraft()`、`getWorkingState()`、`updateDraftWorking()`、`resetDraft()` 和 `commitDraft()`。

这块实现逻辑很清楚：

+ 进入需要试走的阶段时，从 `committed` 派生一份 `snapshot + working`。
+ 界面和查询统一读取 `working`，而不是直接读取 `committed`。
+ 建轨中的每一步都只修改 `working`。
+ 点击“提交草稿”时把 `working` 写回 `committed`。
+ 点击“重置阶段”时把 `working` 恢复成 `snapshot`。

这个机制已经真正接到页面里了，因此当前版本并不是“按钮上写着可以重置，底层其实做不到”，而是能真实回滚整个建轨阶段。

### 5.5 地图、segment、anchor 与线路所有权

这一块是整个规则内核里最像“桌游引擎”的部分。当前地图相关代码主要分布在 `src/map/` 和 `src/rules/trackOwnership.ts`、`src/rules/tokenAnchors.ts`。

它的思路不是只把每块轨道记成“某玩家在某 hex 铺了一块 tile”，而是继续往下拆成：

+ `trackPieces`：地图上真正铺下去的板块。
+ `segments`：把每块板的线路拆成更细的轨道段。
+ `anchors`：把玩家线路所有权绑定到某个确定的 segment。
+ `links`：从 segment 聚合出来的已完成或未完成连接视图。

这样做的好处，是它能支撑比简单“路径数组”更复杂的局面。就算现在 `Urbanization` 还没完全接上，这套底层结构也已经为“线路被打断后重新判定所有权”预留了正确方向。当前版本已经具备这一层底座，因此之后补 `Urbanization` 时不是从零开始。

### 5.6 规则子模块

当前 `src/rules/` 已经拆成若干规则子模块，而不是所有合法性判断都塞进同一个文件：

+ `finance.ts`：现金、即时费用、融资补钱、收入结算相关逻辑。
+ `tilePool.ts`：轨道库存扣减与检查。
+ `trackPlacement.ts`：建轨是否合法、费用是多少。
+ `tokenAnchors.ts`：所有权锚点维护。
+ `trackOwnership.ts`：重建 link 与所有权。
+ `goodsDelivery.ts`：运货候选和得分分配。
+ `routeRanking.ts`：对候选路径做排序。
+ `income.ts`、`scoring.ts`：收入和计分相关逻辑。
+ `auction.ts`：标准版 / 顺位相关的规则准备。

这些模块的组织逻辑，不是按“界面页面”拆，而是按“规则问题域”拆。这样未来调整货运判定时，只需要动 `goodsDelivery` 和相关查询，不必在 UI 组件里到处搜字符串和条件分支。

不过也正因为当前规则模块是按“已经实现的规则问题”拆的，所以能看出哪些东西还没有真正实现成模块化逻辑。比如如果 `City Growth` 与 `Urbanization` 还没有对应的动作与独立规则文件，那就说明它们仍然处在设计预留阶段，而不是完成阶段。

### 5.7 查询层：给 UI、Bot、测试的只读视图

`src/queries/` 负责的不是改状态，而是从当前状态中整理出“页面或 Bot 真正需要读取的结果”。这是当前架构里另一个非常重要的分层。

当前查询层已经按领域拆开，例如：

+ `actionTileQueries.ts`
+ `trackQueries.ts`
+ `deliveryQueries.ts`
+ `turnQueries.ts`
+ `tilePoolQueries.ts`
+ `previews.ts`
+ `selectors.ts`
+ `explanations.ts`

例如，页面不会自己枚举地图 hex 去猜哪些地方能铺轨，而是调用 `getBuildableHexIds()`；也不会自己穷举货运路径，而是调用 `getRankedDeliveryCandidatesForSource()`；也不会自己决定当前阶段如何描述，而是通过 `selectPhaseSummary()` 获取摘要。

这一层的存在，让 UI 从“逻辑堆场”变成了“状态消费者”。也正是因为有这一层，Bot、测试和 UI 才能复用同一套合法动作与候选方案查询，而不会三套口径各写一遍。

### 5.8 规则解释模型

当前规则解释边界已经收到了方案要求的方向：`game-core` 不再直接输出最终面向用户的整段中文文案，而是先在 `src/queries/explanations.ts` 里输出结构化 explanation model；Web 层再通过 presenter 把它变成当前页面要显示的中文说明。

这件事的意义不只是“架构更优雅”。它直接决定了之后如果你要把解释文本改成更像教程、更像速查卡、或者支持不同文体时，不需要动规则引擎本身。也就是说，当前仓库已经把“规则为什么成立”和“界面怎么说给人听”拆开了。

### 5.9 Bot、回放与序列化

当前 Bot 层已经有 `Bot` 接口、`RandomBot` 实现和 `botTurn.ts`。现在的 Bot 并不追求聪明，它只负责在自己回合里读取当前合法动作并执行一个简单动作。这刚好符合当前项目的教学定位：让真人只操作自己一个阵营，其余玩家用最弱托管补齐桌面人数。

回放在 `src/replay/replay.ts`。它当前采用的是“每次状态推进后记录一帧快照”的轻量方式，而不是复杂的时间旅行系统。每帧会记录标签、回合、阶段、当前行动者和一份会话快照，页面随后可以基于这些帧恢复到某一时刻。

序列化在 `src/serialization/`。它已经不是“直接 JSON.stringify 会话”这么简单，而是有显式的 schema version 和 `migrations/`。这意味着当前仓库已经开始按“长期维护”的方式处理存档格式变化：旧存档先迁移，再反序列化，而不是版本一变就报错。

## 六、`apps/web`：本地浏览器学习壳

如果说 `game-core` 负责“规则真相”，那 `apps/web` 负责的就是“把这套真相变成一个人可以实际使用的本地页面”。

### 6.1 应用主入口与 Provider

当前页面主入口是：

+ `src/app/main.tsx`
+ `src/app/App.tsx`
+ `src/app/providers/GameSessionProvider.tsx`
+ `src/pages/game/GamePage.tsx`

其中最关键的是 `GameSessionProvider.tsx`。它是 Web 层的会话中枢，负责：

+ 创建默认对局。
+ 管理 `setup / session / replayFrames / savedGames / notice`。
+ 计算当前工作态、当前玩家、阶段摘要、真人玩家身份和是否轮到真人。
+ 暴露新开一局、保存、载入、删除存档、恢复回放帧等命令。

这意味着页面组件本身并不直接操纵浏览器存储和会话初始化，而是通过 Provider 拿到一套已经整理好的运行上下文。这个分层让页面树变得可维护很多。

### 6.2 页面主装配逻辑

当前真正把各个面板接在一起的是 `src/pages/game/GamePage.tsx`。它负责做几件事：

+ 从 Provider 取出当前会话和派生信息。
+ 调用 `game-core` 的查询函数，生成当前阶段所需的 buildable hex、朝向选项、建轨预览、可运货源、候选运输方案、规则解释等派生数据。
+ 维护页面交互需要的局部选择状态，例如“当前选中的轨道板”“当前选中的 hex”“当前选中的货物源”“当前选中的运输候选”。
+ 把这些数据分发给具体面板组件。

这个页面不是简单把组件排个版，而是承担了“把 engine session 翻译成当前 UI 工作台”的装配职责。也就是说，页面里最核心的逻辑不是规则计算本身，而是“当前阶段该显示哪些工作台、这些工作台之间如何联动”。

### 6.3 会话命令封装

`src/features/session-control/useSessionCommands.ts` 把页面最常用的操作都封成了一组 hook 命令，例如：

+ 选择行动牌。
+ 放置轨道。
+ 结束建轨。
+ 重置建轨草稿。
+ 提交建轨草稿。
+ 执行运货。
+ 升级机车。
+ 跳过货运。
+ 结算当前阶段。
+ 推进 Bot 直到轮到真人。
+ 调试单步和调试十步。

这层的价值，在于让页面按钮只关心“什么时候触发哪个命令”，而不必每个按钮都自己拼一段 `applyAction(session, ...)`。这明显降低了 UI 层逻辑重复。

### 6.4 具体面板是如何分工的

当前 Web 层已经按功能拆出了相对清晰的 feature 模块，而不是所有 JSX 都堆在 `GamePage`。

重要模块包括：

+ `features/board/`：地图棋盘、hex 几何、tile 形状、建轨面板、轨道板选择面板。
+ `features/delivery/`：货物源选择、候选运输方案、运输预览。
+ `features/session-control/`：阶段信息、行动牌、顺位面板、命令 hook。
+ `features/players/`：玩家状态展示。
+ `features/supply/`：轨道库存和 Goods Supply 展示。
+ `features/rule-help/`：规则解释、即时提示、非法动作提示、presenter。
+ `features/save-load/`：浏览器本地存档仓库和对应面板。
+ `features/replay/`：回放帧时间线。
+ `features/logs/`：动作历史与日志。
+ `features/game-setup/`：新对局设置和默认玩家名生成。

它们之间的连接方式是：`GamePage` 负责集中拿数据和命令，每个 feature panel 只处理某一块清晰的显示职责。当前仓库已经基本摆脱了“组件里顺手写规则、规则里顺手写界面文案”那种难维护状态。

### 6.5 规则解释与存档为什么放在 Web 层

当前解释系统采取的是“core 产出结构化原因，Web presenter 输出最终文案”的方式；存档系统则采取“core 只负责序列化契约，Web repository 负责浏览器 localStorage”的方式。

这两条边界都很合理：

+ 解释文本属于展示策略，应该允许页面决定语言风格和呈现形式。
+ `localStorage` 属于浏览器能力，不应该污染纯规则内核。

当前 `browserSaveRepository.ts` 已经把这件事处理成了一个独立 Web 仓库模块，并且对宿主 `localStorage` 能力做了兼容适配和内存回退。也就是说，当前 Web 层不只是“有个按钮能保存”，而是已经把本地存档作为一项明确的基础设施收起来了。

## 七、测试、验证与工程约束

当前仓库的工程部分已经超出了“随便能跑就行”的程度，开始有比较严肃的长期维护约束。

### 7.1 测试层次

`packages/game-core/tests/` 当前已经分成：

+ `scenarios/`：偏规则场景测试。
+ `workflows/`：偏工作流和集成流程测试。
+ `invariants/`：偏系统不变量和迁移链测试。
+ `helpers/`：测试辅助工具。

此外，`apps/web` 也有页面层与存档层的测试。

这说明当前测试并不是只覆盖“函数 A 输入 1 输出 2”，而是已经开始覆盖更靠近真实使用的层面，比如建局、阶段推进、序列化迁移、页面载入和存档行为。

### 7.2 当前验证链

当前根目录脚本已经支持并常态化运行：

+ `npm run typecheck`
+ `npm run lint`
+ `npm run check:deps`
+ `npm run check:boundaries`
+ `npm test`
+ `npm run build`
+ `npm run smoke`

这条链的意义分别是：

+ `typecheck` 保证跨包类型一致。
+ `lint` 和 `check:boundaries` 保证导入边界不被打穿。
+ `check:deps` 保证包之间依赖方向不倒置。
+ `test` 保证规则、工作流和页面行为。
+ `build` 保证工作区仍可构建。
+ `smoke` 保证“至少能创建一局并走通一条基础主路径”。

从系统说明的角度看，这意味着当前仓库已经不只是“代码存在”，而是“围绕当前代码存在一套持续验证机制”。

## 八、当前版本的典型运行路径

如果把当前系统从启动到用户操作串成一条主线，它大致是这样工作的：

1. 浏览器启动 `apps/web`。
2. `GameSessionProvider` 基于默认设置，通过 `game-content` 取地图、规则集和内容目录，再调用 `game-core/createGame()` 创建一局默认基础版东北美洲对局。
3. `GamePage` 读取当前会话，通过查询层计算当前阶段摘要、合法落点、预览和候选方案。
4. 真人玩家在页面上点击按钮或地图，`useSessionCommands` 把这些交互翻译成显式 `GameAction`。
5. `applyAction()` 把动作分派到对应 handler，更新 `committed` 或 `draft`，并追加动作历史。
6. Provider 收到新会话后，刷新 `notice`、日志与回放帧。
7. 页面重新渲染，地图、工作台、规则解释、玩家面板和公共区同步更新。
8. 如有需要，用户还可以把当前会话序列化到本地存档，之后再重新载入。

这条链说明当前仓库最核心的实现逻辑，确实已经从“静态设计文档”变成了“真实可运行的软件结构”。

## 九、当前实现与最初总方案的关系

如果把现在的实现和 [0404_Codex_Steam数字化实现方案.md](0404_Codex_Steam数字化实现方案.md) 对照，应该这样理解两者的关系：

+ 总方案里关于“规则内核与 UI 分离、使用 React + TypeScript + SVG、本地浏览器单页应用、segment + anchor + link、草稿态、候选运输方案、Bot、规则解释、存档回放、长期维护工具链”的大方向，当前已经基本落地。
+ 总方案里关于“基础版和标准版都完整闭合、更多地图完整可玩、`City Growth` / `Urbanization` 等规则书主行动全部接进 UI 与动作层、真正能完整打完一局 Steam”的最终目标，当前还没有全部完成。

也就是说，当前仓库已经完成了“正确的骨架、正确的主要模块、正确的主要工作流”，但还没有完成“规则书层面的全覆盖收口”。这也是为什么它现在更适合被称为**当前实现版系统**或**学习型数字原型**，而不是**完整电子版 Steam**。

## 十、结论

当前项目已经不是早期的“只有设计文档和几个演示组件”的状态了。它已经有清晰的工作区结构、可以启动的本地页面、可运行的规则内核、明确的状态与动作模型、真人与 Bot 混合的主路径、建轨草稿、运货候选、规则解释、本地存档、回放、测试和工程边界检查。从软件结构上看，这已经是一个相当完整的教学型桌游系统雏形。

但如果问题是“是否已经按照规则书，写出了一个完整可跑通的电子版 Steam 桌游”，当前更准确的回答仍然是：**还没有完全完成**。它已经完成了最有价值、最难返工、最能支撑后续扩展的主体部分；要达到“完整电子版”的标准，还需要继续把规则书中尚未接入动作层和页面层的主行动、模式差异与完整流程补齐。

因此，这份文档的定位不是“宣布项目完成”，而是给当前仓库一个清晰、系统、诚实的技术快照：**现在它已经实现到了哪里，它为什么能跑，它为什么还没有完全结束，后续真正要补的又是什么。**
